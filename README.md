# インターン選考管理アプリ — PWA対応 要件・実装ガイド

## 1. 概要

本アプリは、応募したインターン先企業の選考状況、締め切り、面接日時、希望度、企業マイページURL、ログインID、パスワードを一元管理するための**ローカルファーストPWA（Progressive Web App）**です。

利用者は企業名と卒業年度を指定して、登録済みの企業マイページURLを確認できます。また、ID・パスワードは明示的な操作でコピーでき、パスワードは通常は伏字で扱います。

> **重要**：本アプリは認証情報を扱います。パスワードを平文のまま保存・ログ出力・通知表示してはいけません。

---

## 2. 目的

- 応募先企業の情報を一覧で管理する。
- 締め切り・面接予定・選考状況を把握しやすくする。
- 企業マイページURL、ID、パスワードへのアクセスを簡単にする。
- ネットワークが不安定な環境でも、登録済みデータを確認・編集できるようにする。
- スマートフォンおよびPCで、アプリのようにインストールして利用できるようにする。

---

## 3. 非目標

以下は対象外です。

- 企業のインターン情報、締め切り、選考結果、マイページURLを外部サイトから自動取得すること。
- 応募フォームの自動入力や自動送信。
- 求人サイト・企業サイトのスクレイピング。
- パスワードを他ユーザーと共有する機能。
- 1企業に複数の企業マイページURLを登録すること。

---

## 4. 主な機能

### 4.1 企業・選考情報の管理

- 企業名、卒業年度、応募状況、希望度の登録・編集・削除
- 締め切りおよび面接日時の登録
- メモの登録
- 企業一覧の表示、検索、絞り込み、並び替え
- 締め切り順・面接日時順での確認

### 4.2 企業マイページ情報の管理

- 企業エントリごとに企業マイページURLを最大1件登録
- 企業名と卒業年度からURLを検索・表示
- URLをブラウザで開く
- URLをクリップボードへコピー
- URL未登録時の登録案内

### 4.3 認証情報の管理

- マイページID・パスワードの登録・更新
- IDのワンクリックコピー
- パスワードのワンクリックコピー
- パスワードの伏字表示と明示操作による一時表示
- パスワードの暗号化保存

## 5. PWA機能要件

### 5.1 インストール可能なアプリ

アプリはWeb App Manifestを提供し、対応ブラウザでホーム画面またはアプリ一覧へインストールできること。

| ID | 要求 |
|---|---|
| PWA-FR-01 | `manifest.webmanifest` を提供すること。 |
| PWA-FR-02 | アプリ名、短縮名、テーマカラー、背景色、起動URL、表示モード、アイコンを定義すること。 |
| PWA-FR-03 | `display: standalone` を設定し、インストール後にアプリ風の表示で起動できること。 |
| PWA-FR-04 | 192×192、512×512以上を含む複数サイズのアプリアイコンを提供すること。 |
| PWA-FR-05 | インストール可能な環境では、利用者の明示操作によりインストール案内を表示できること。 |
| PWA-FR-06 | インストールできない環境では、ブラウザのホーム画面追加方法を案内できること。 |

### 5.2 オフライン対応

| ID | 要求 |
|---|---|
| PWA-FR-07 | Service Workerを登録し、アプリシェルをオフラインで起動できること。 |
| PWA-FR-08 | 静的ファイル（HTML、CSS、JavaScript、フォント、アイコン）はキャッシュして利用可能にすること。 |
| PWA-FR-09 | 登録済みの企業・選考情報は、オフライン時にも閲覧できること。 |
| PWA-FR-10 | 企業情報、選考情報、URL、ID、暗号化済みパスワードはIndexedDBへ永続化できること。 |
| PWA-FR-11 | オフライン中の追加・更新・削除はローカルに反映できること。 |
| PWA-FR-12 | サーバー同期を実装する場合、オフライン中の変更は同期キューへ追加し、復帰後に再送できること。 |
| PWA-FR-13 | 同期機能を実装しないローカル専用版では、データが端末内に保存されることを画面上で明示すること。 |

### 5.3 更新管理

| ID | 要求 |
|---|---|
| PWA-FR-14 | Service Workerの新しいバージョンが利用可能になったとき、利用者に更新可能であることを通知すること。 |
| PWA-FR-15 | 利用者が「更新する」を選択した場合、キャッシュを安全に切り替え、アプリを再読み込みすること。 |
| PWA-FR-16 | 更新前の未保存入力が失われないよう、フォームの下書き保存または離脱確認を行うこと。 |

### 5.4 通知・リマインダー

| ID | 要求 |
|---|---|
| PWA-FR-17 | 利用者は締め切り・面接日時の通知設定を企業エントリごとに登録できること。 |
| PWA-FR-18 | 通知権限は利用者の明示操作後にのみ要求すること。 |
| PWA-FR-19 | 通知には企業名、予定種別、日時のみを表示し、URL・ID・パスワードを含めないこと。 |
| PWA-FR-20 | 通知をタップすると、該当企業の詳細画面を開けること。 |
| PWA-FR-21 | Push APIやバックグラウンド機能が利用できない環境では、アプリ起動時のアプリ内通知へフォールバックすること。 |

### 5.5 バックアップ・復元

| ID | 要求 |
|---|---|
| PWA-FR-22 | 利用者は企業・選考情報をエクスポートできること。 |
| PWA-FR-23 | 認証情報を含むバックアップは、暗号化ファイルとして出力すること。 |
| PWA-FR-24 | 利用者は暗号化バックアップを選択し、復元できること。 |
| PWA-FR-25 | CSV出力を提供する場合、パスワードは初期設定では含めないこと。 |

## 6. 推奨アーキテクチャ

```text
Browser / Installed PWA
├── UI層
│   ├── 企業一覧
│   ├── 企業登録・編集
│   ├── URL確認
│   ├── 認証情報表示・コピー
│   └── 設定・バックアップ
├── Application層
│   ├── CompanyEntryController
│   ├── CredentialController
│   ├── NotificationScheduler
│   └── SyncQueueService（同期型の場合）
├── Domain層
│   ├── CompanyEntry
│   ├── SelectionRecord
│   ├── PortalCredential
│   ├── Deadline
│   └── Interview
├── Storage層
│   ├── IndexedDB
│   ├── EncryptedCredentialStore
│   ├── Cache Storage
│   └── Service Worker
└── Optional Backend
    ├── 認証
    ├── 暗号化同期
    ├── Push通知
    └── 監査ログ
```

### 6.1 推奨データ保存先

| データ | 保存先 | 方針 |
|---|---|---|
| 企業・選考情報 | IndexedDB | オフラインでも検索・表示・編集できるようにする |
| URL・ID | IndexedDB | ローカル保存。同期する場合は転送時・保存時とも保護する |
| パスワード | IndexedDB（暗号文のみ） | Web Crypto APIで暗号化し、平文は必要時のみメモリ上で扱う |
| アプリ静的ファイル | Cache Storage | Service Workerで管理する |
| 同期待ち操作 | IndexedDB | オフライン時の更新をキューイングする |

## 7. PWAキャッシュ戦略

| 対象 | 戦略 | 理由 |
|---|---|---|
| HTML・JavaScript・CSS・アイコン | Precache / Cache First | アプリをオフラインでも起動可能にするため |
| APIの企業一覧・詳細取得 | Network First + Offline Fallback | オンラインでは新しい情報を優先し、オフラインではローカル情報を表示するため |
| 利用者の入力データ | IndexedDBを正とする | キャッシュ削除による情報喪失を避けるため |
| 通知用データ | Network First | 古い締め切り通知を避けるため |
| 外部マイページURL | キャッシュしない | 企業サイトの認証状態や最新性をアプリ側で保持しないため |

### 7.1 Service Workerの注意点

- Service WorkerはHTTPS環境で配信する。
- `skipWaiting()` は自動実行せず、更新確認後に実行する。
- 古いキャッシュは新バージョン適用後に削除する。
- 企業マイページの外部サイト、認証画面、ID・パスワードの平文レスポンスはキャッシュしない。

## 8. セキュリティ要件

### 8.1 認証情報の保護

| ID | 要求 |
|---|---|
| SEC-01 | パスワードは平文でIndexedDB、Local Storage、Cache Storage、ログ、通知、分析イベントに保存しないこと。 |
| SEC-02 | パスワードはWeb Crypto API等を用いて暗号化し、暗号文のみを永続保存すること。 |
| SEC-03 | 暗号鍵は利用者のPIN・パスフレーズまたはWebAuthn等により保護すること。 |
| SEC-04 | 平文パスワードは、表示またはコピーの直前にのみ復号し、処理完了後はメモリ上に保持し続けないこと。 |
| SEC-05 | パスワードは通常伏字で表示し、表示操作は利用者の明示操作でのみ許可すること。 |
| SEC-06 | パスワードのコピーは明示操作でのみ実行し、コピー完了後はクリップボードに機密情報が残る可能性を利用者へ案内すること。 |
| SEC-07 | バックアップに認証情報を含める場合は、利用者指定のパスフレーズで暗号化すること。 |
| SEC-08 | URL・メモ・企業名など、利用者入力値を表示する際はXSS対策としてHTMLエスケープまたは安全なテンプレート描画を行うこと。 |
| SEC-09 | Content Security Policy（CSP）を設定し、許可していないスクリプトの実行を防止すること。 |
| SEC-10 | 外部企業マイページを開くリンクには、安全なリンク設定を適用すること。 |

### 8.2 端末共有への配慮

- アプリ起動時または一定時間の無操作後にロックする。
- ロック解除にはPIN、端末認証、生体認証、またはWebAuthnを利用できる構成が望ましい。
- アプリ切替・バックグラウンド遷移時に、パスワード平文を画面から隠す。
- Push通知やアプリ通知にID・パスワードを含めない。

> ブラウザ内の暗号化だけでは、端末自体が侵害されている場合や同一オリジン上の悪意あるスクリプト実行まで完全には防げません。CSP、依存ライブラリ管理、HTTPS、XSS対策を必須とします。

## 9. UI・UX要件

### 9.1 主な画面

1. **企業一覧画面**
   - 企業名、卒業年度、応募状況、希望度、締め切り、面接日時を一覧表示
   - URLを開く・コピーする
   - ID・パスワードをコピーする
   - パスワード表示切替
   - 検索、絞り込み、並び替え

2. **企業登録・編集画面**
   - 企業・選考情報、URL、認証情報、メモの入力
   - 応募状況・希望度は選択式
   - 締め切り・面接日時は日付時刻ピッカー

3. **URL確認画面**
   - 企業名・卒業年度で検索
   - URLを表示、開く、コピー
   - URL未登録の場合は登録画面へ誘導

4. **PWA設定画面**
   - アプリのインストール案内
   - オフライン状態の表示
   - 更新の適用
   - 通知設定
   - アプリロック設定
   - バックアップ・復元

### 9.2 オフライン表示

- オフライン中であることを常時識別できる表示を行う。
- 保存成功時は「この端末に保存済み」、同期済みの場合は「クラウド同期済み」など、保存状態を区別して表示する。
- 同期に失敗した操作は、再試行可能なキューとして表示する。

## 10. データモデル

### 10.1 主要エンティティ

```text
User
└── CompanyEntry（0..*）
    ├── Company（1）
    ├── SelectionRecord（1）
    │   ├── Deadline（0..1）
    │   └── Interview（0..*）
    └── PortalCredential（0..1）
```

### 10.2 主要データ項目

| エンティティ | 主な項目 |
|---|---|
| CompanyEntry | entryId、userId、companyId、graduationYear、createdAt、updatedAt |
| Company | companyId、companyName |
| SelectionRecord | applicationStatus、preferenceLevel、memo |
| Deadline | state、dueAt |
| Interview | state、scheduledAt、interviewType、locationOrMeetingUrl、note |
| PortalCredential | portalUrl、loginId、passwordCiphertext、updatedAt |
| SyncQueueItem | queueId、operationType、entityId、payloadCiphertext、createdAt、retryCount |

### 10.3 制約

- `User × Company × graduationYear` は一意。
- `CompanyEntry` ごとの `PortalCredential` は0件または1件。
- URLは `https://` または `http://` で始まる形式のみ登録可能。
- パスワードは平文保存禁止。

## 11. オフライン・同期フロー

### 11.1 ローカル専用モード

1. 利用者が企業情報を登録・更新する。
2. 入力検証後、IndexedDBへ保存する。
3. オフラインでも一覧・検索・編集・コピーを実行できる。
4. ブラウザデータを削除した場合、データが失われる可能性があるため、定期バックアップを案内する。

### 11.2 同期対応モード

1. 利用者が変更を保存する。
2. まずIndexedDBへ保存する。
3. オンライン時はバックエンドへ暗号化済みデータを同期する。
4. オフライン時は同期キューへ追加する。
5. 通信復帰後、キューを順に送信する。
6. 競合が発生した場合は、更新日時だけで自動上書きせず、利用者に競合解決を促す。

## 13. 受け入れ基準（PWA追加分）

### AC-PWA-01：インストール

**前提**：対応ブラウザでアプリへアクセスしている。  
**操作**：利用者がインストール操作を行う。  
**期待結果**：ホーム画面またはアプリ一覧にアプリが追加され、単独ウィンドウとして起動できる。

### AC-PWA-02：オフライン起動

**前提**：利用者が一度オンラインでアプリを起動済みである。  
**操作**：ネットワークを切断してアプリを開く。  
**期待結果**：アプリシェルと登録済み企業情報が表示される。

### AC-PWA-03：オフライン編集

**前提**：ネットワークが切断されている。  
**操作**：利用者が企業情報を追加または編集して保存する。  
**期待結果**：変更内容が端末内のデータストアへ保存され、一覧に反映される。

### AC-PWA-04：認証情報の秘匿

**前提**：企業にIDとパスワードが登録されている。  
**操作**：利用者が企業一覧を開く。  
**期待結果**：パスワードは伏字で表示され、平文のパスワードはストレージおよび画面に常時保存・表示されない。

### AC-PWA-05：明示的なコピー

**前提**：利用者がアプリロックを解除済みである。  
**操作**：IDまたはパスワードのコピー操作を行う。  
**期待結果**：選択した情報だけがコピーされ、コピー成功メッセージが表示される。

### AC-PWA-06：更新通知

**前提**：新しいアプリバージョンが配信されている。  
**操作**：利用者がアプリを利用する。  
**期待結果**：更新可能なことが表示され、利用者の承認後に安全に更新できる。

### AC-PWA-07：通知権限

**前提**：利用者が通知をまだ許可していない。  
**操作**：利用者が通知設定画面で「通知を有効にする」を選択する。  
**期待結果**：その操作を契機としてのみブラウザの通知権限を要求する。

## 15. 開発時の推奨事項

- PWAキャッシュ管理には、Workboxなどの実績ある仕組みを利用する。
- ローカルDBにはIndexedDBを利用し、アプリ側ではDexie等のラッパーを採用すると保守しやすい。
- 認証情報を同期する場合は、サーバー側でも暗号化・アクセス制御・監査を実装する。
- 認証情報を扱うため、依存パッケージの脆弱性確認、CSP、XSS対策、HTTPSをリリース条件に含める。
- オフライン時の利用を前提に、ネットワーク障害をエラーではなく状態としてUIに表現する。

## 16. リリース判定（最低条件）

以下をすべて満たした場合に、PWA対応版のリリース候補とする。

1. アプリがインストールできる。
2. 一度利用した後、オフラインで起動できる。
3. オフラインで企業・選考情報を登録・編集・確認できる。
4. ID・パスワードを明示操作でコピーできる。
5. パスワードが平文のまま永続保存されない。
6. Service Worker更新時に利用者データが失われない。
7. 通知には認証情報を含めない。
8. ブラウザのデータ削除に備えて、バックアップ手段または注意表示がある。
